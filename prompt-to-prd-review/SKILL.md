# Prompt to PRD Review

## Purpose

Convert clarified product requirement materials into a PRD, refine it through frontend and backend review, and save the PRD only after both reviewer perspectives approve the current draft.

This skill starts after the product intent is mostly clarified, whether it came from requirement-exploration, a direct feature description, an existing PRD update, or multiple source materials. Do not restart broad discovery. Preserve the clarified requirement, make unresolved items visible, handle new requirements and existing-project iterations differently, and treat reviewer approval as a gate before writing the final file.

## Use This Skill When

- The user provides a prompt generated by `requirement-exploration`.
- The user says requirement exploration is complete and now needs a PRD.
- The user provides an existing PRD plus a simple change description and wants an updated/new PRD.
- The user directly describes a sufficiently clear feature and asks for a PRD file with review/approval before saving.
- The user wants frontend and backend reviewers or subagents to check the PRD before finalizing.
- The user wants the PRD written into a file only after review approval.
- The user is adding or changing a requirement in an existing project and wants review against current frontend/backend reality.
- The user provides several requirement materials and wants them consolidated into a PRD.

## Do Not Use This Skill When

- The requirement is still vague and needs initial clarification first.
- The user only wants the downstream prompt, not a PRD.
- The user asks for technical design, API design, database schema, or task breakdown instead of PRD generation.
- The user explicitly wants a quick one-shot PRD draft without review.

If the input is too incomplete to produce a trustworthy PRD, ask one focused blocking question instead of inventing missing product facts.

## Core Responsibilities

- Treat the requirement-exploration output as the authoritative product source unless newer user instructions override it.
- Classify the work as a new requirement, existing-project iteration, or unclear mode.
- For existing-project iterations, gather enough current-project context to avoid writing an incompatible PRD.
- Draft a structured PRD with explicit assumptions, open questions, scope, business rules, state/data behavior, acceptance criteria, and compatibility constraints when applicable.
- Bring in frontend and backend reviewers only when the PRD is coherent enough to review.
- Iterate on reviewer feedback until both perspectives approve or a user decision is needed.
- Save only the approved current PRD and append a review/version record.

## Success Standard

This skill is successful when:

- The PRD faithfully reflects the explored requirement and preserves uncertainty instead of guessing.
- Multi-source inputs are merged with a clear source-priority rule.
- Existing-project iterations reflect current project reality and compatibility risks.
- Frontend and backend review feedback leads to concrete PRD improvements.
- Blocking open questions stop the flow; non-blocking questions are recorded.
- Both reviewer perspectives approve the final PRD before it is saved.
- The saved Markdown file records the PRD, review outcome, project context, and version/change summary.
