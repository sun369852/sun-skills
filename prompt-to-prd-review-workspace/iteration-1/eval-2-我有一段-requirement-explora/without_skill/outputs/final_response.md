# 处理结论

当前不建议直接生成并保存最终 PRD。

阻塞点是：**用户改名后，历史已发证书是否允许按新姓名重新生成**。这不是一个文案细节，而是证书真实性、历史审计、重发规则、编号策略和作废/失效语义的核心规则。如果在这个问题未定的情况下落 PRD，后续前后端、数据表、权限、审计日志和运营流程都可能按错误方向实现。

# 需要确认的核心问题

请业务方确认以下规则：

**用户改名后，历史已发证书是否允许使用新姓名重新生成？**

建议在下面两种方案中选择一种：

1. **允许重新生成**
   - 历史证书可以因用户姓名变更触发重发。
   - 原证书应进入失效或作废状态，并保留历史记录。
   - 新证书使用新姓名生成，并生成新的证书文件。
   - 需要明确证书唯一编号是否复用原编号，还是生成新编号。
   - 需要记录重发原因为“用户姓名变更”。

2. **不允许重新生成**
   - 已发证书保持发证当时姓名，作为历史凭证。
   - 用户改名只影响后续新发证书。
   - 运营后台可查看用户当前姓名与证书姓名不一致的情况，但不得直接改写历史证书。
   - 如需特殊处理，只能通过手动作废并重新发放的独立流程处理。

# 为什么该问题会阻塞 PRD

该规则会影响至少以下 PRD 内容：

- 证书状态流转：改名重发是否属于“重发”、是否自动作废旧证书。
- 唯一编号规则：新姓名生成的证书是否沿用原编号。
- 历史记录口径：历史证书展示发证时姓名还是当前姓名。
- 后台操作权限：运营是否可以因改名主动重发。
- 审计要求：是否记录姓名变更前后值、操作者、原因和时间。
- 用户端展示：用户能否看到旧证书、失效证书、新证书。
- 合规和真实性：证书作为培训完成凭证时，是否允许事后改写身份信息。

# 建议下一步

先补充确认这一条业务决策，再生成 PRD。

推荐优先采用的产品规则是：**不自动改写历史已发证书；如确需按新姓名出证，必须通过明确的作废并重发流程，保留旧证书记录和审计日志，新证书生成新编号**。

该方案更利于保持历史真实性，也和“作废后原证书状态变失效；重发保留历史记录”的现有要求一致。
