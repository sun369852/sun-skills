# PRD: 订单售后模块增加部分退款原因标签

## 1. 背景

运营在审核部分退款时，需要结构化记录退款原因，便于后续查询、统计和导出分析。当前需求仅面向后台运营管理场景，不影响 C 端用户退款体验。

## 2. 目标

- 后台运营审核部分退款时，必须选择一个原因标签。
- 后台退款列表支持按原因标签筛选。
- 后台退款详情展示原因标签和选择该标签的操作人。
- 后台导出报表包含原因标签字段。
- 新字段仅后台可见，C 端用户不展示、不感知。
- 沿用现有售后审核权限，不新增角色或权限。

## 3. 非目标

- 不补录历史退款单的原因标签。
- 不新增 C 端展示、通知、消息、订单详情字段。
- 不新增角色、权限点或组织架构配置。
- 本期不实现“已审核通过后修改原因标签”的能力；该项作为开放问题记录。

## 4. 用户与场景

主要用户：后台运营/售后审核人员。

核心场景：

1. 运营打开待审核的部分退款单。
2. 运营判断退款原因，从固定标签中选择一个。
3. 运营提交审核通过。
4. 后续运营可在退款列表中按原因标签筛选，并在详情页或导出报表中查看标签。

## 5. 业务规则

### 5.1 原因标签枚举

系统内置以下固定标签：

| 标签值 | 展示文案 | 说明 |
| --- | --- | --- |
| `out_of_stock` | 商品缺货 | 因商品库存不足或无法发货导致部分退款 |
| `damaged_goods` | 商品破损 | 因商品破损、瑕疵等导致部分退款 |
| `shipping_difference` | 运费差额 | 因运费调整、运费补差、运费退差导致部分退款 |
| `service_compensation` | 客服补偿 | 因客服安抚、体验补偿等导致部分退款 |
| `other` | 其他 | 以上原因均不适用 |

### 5.2 适用范围

- 仅适用于后台审核“部分退款”时。
- 全额退款、仅退货、换货、补发等非部分退款流程不强制选择该标签，除非现有系统将其统一归入部分退款审核。
- 历史退款单不补录，新字段为空。

### 5.3 必填规则

- 新需求上线后，后台运营审核通过部分退款时，原因标签为必填。
- 未选择原因标签时，审核通过提交应被拦截，并提示运营选择原因标签。
- 审核拒绝是否需要选择标签：不需要，本需求只要求“审核部分退款时选择原因标签”，当前版本按审核通过场景强制。

### 5.4 可见性规则

- 后台退款列表、详情、导出可展示原因标签。
- C 端用户不可见，包括但不限于 C 端退款详情、订单详情、退款进度、消息通知、客服对客话术字段。
- 若 C 端接口与后台接口复用数据模型，接口响应必须显式过滤该字段。

### 5.5 权限规则

- 选择原因标签的权限沿用现有售后审核权限。
- 拥有现有售后审核权限的用户，可在审核部分退款时选择原因标签。
- 无现有售后审核权限的用户，不应因为本需求获得任何新增操作能力。
- 不新增角色、不新增权限点、不新增菜单权限配置。

## 6. 功能需求

### 6.1 部分退款审核

在后台部分退款审核页面或审核弹窗中增加“部分退款原因标签”选择控件。

要求：

- 控件类型：单选，下拉框或单选按钮组均可，遵循现有后台表单风格。
- 选项固定为：商品缺货、商品破损、运费差额、客服补偿、其他。
- 审核通过时必填。
- 提交成功后记录标签、操作人、记录时间。
- 操作人为提交审核通过并选择标签的后台用户。

异常与提示：

- 未选择标签时，不允许审核通过。
- 提示文案建议：`请选择部分退款原因标签`。

### 6.2 退款列表筛选

后台退款列表增加“原因标签”筛选项。

要求：

- 默认值为全部。
- 可按五个固定标签筛选。
- 选择具体标签时，仅返回该标签匹配的退款单。
- “全部”包含无标签的历史退款单。
- 历史退款单不需要提供“无标签”专项筛选，除非现有列表已有类似空值筛选模式。
- 列表列是否展示原因标签：建议展示，便于筛选结果核对；若列表列宽受限，可仅在详情展示，但筛选结果必须准确。

### 6.3 退款详情展示

后台退款详情页展示以下字段：

| 字段 | 展示规则 |
| --- | --- |
| 部分退款原因标签 | 有值展示标签文案；无值展示 `--` |
| 标签操作人 | 有值展示操作人名称或账号；无值展示 `--` |
| 标签记录时间 | 建议展示；有值展示时间，无值展示 `--` |

若现有详情页已有审核信息区，优先放在审核信息区；否则放在退款基础信息或售后处理信息区。

### 6.4 导出报表

退款导出报表增加字段：

| 字段名 | 内容 |
| --- | --- |
| 部分退款原因标签 | 标签展示文案；无值为空 |

要求：

- 导出筛选条件与列表保持一致。
- 若用户按原因标签筛选后导出，导出结果应与筛选结果一致。
- 历史退款单或非适用单据该字段为空。
- 不要求导出操作人，除非现有报表已导出审核操作人并适合复用。

## 7. 数据与接口建议

### 7.1 数据字段

建议在退款/售后单主表或部分退款审核记录表增加以下字段，具体落表以现有模型为准：

| 字段 | 类型建议 | 是否可空 | 说明 |
| --- | --- | --- | --- |
| `partial_refund_reason_tag` | string/enum | 是 | 原因标签枚举值 |
| `partial_refund_reason_operator_id` | bigint/string | 是 | 选择标签的后台用户 ID |
| `partial_refund_reason_operator_name` | string | 是 | 选择标签时的后台用户展示名快照 |
| `partial_refund_reason_set_at` | datetime | 是 | 标签记录时间 |

兼容性要求：

- 字段必须允许为空，以兼容历史数据。
- 不进行历史数据补录。
- 新上线后的部分退款审核通过记录必须写入标签与操作人信息。

### 7.2 后台接口

部分退款审核提交接口：

- 新增入参：`partialRefundReasonTag`。
- 审核通过且退款类型为部分退款时，该字段必填并校验枚举合法性。
- 后端从登录态/操作上下文记录操作人，不信任前端传入的操作人。

退款列表查询接口：

- 新增查询参数：`partialRefundReasonTag`。
- 参数为空表示不过滤。
- 参数为具体枚举值时，按对应标签过滤。

退款详情接口：

- 后台详情响应增加原因标签、操作人、记录时间字段。

导出接口：

- 支持原因标签筛选参数。
- 导出结果增加“部分退款原因标签”列。

C 端接口：

- 不新增该字段。
- 如底层 DTO 复用，必须过滤原因标签和操作人字段。

## 8. 状态与历史数据

- 历史退款单保持无标签状态。
- 后台列表“全部”可查到历史退款单。
- 后台详情中历史退款单显示 `--`。
- 导出历史退款单时“部分退款原因标签”为空。
- 不触发历史数据迁移补录任务。

## 9. 开放问题

| 问题 | 当前处理 | 影响 |
| --- | --- | --- |
| 是否允许运营修改已审核通过的标签？ | 未定。本期不提供修改入口和修改接口，仅记录为后续决策。 | 若后续允许修改，需要补充权限、操作日志、修改原因、导出取值规则、审计追踪。 |

## 10. 验收标准

1. 拥有售后审核权限的运营审核通过部分退款时，必须选择且只能选择一个原因标签。
2. 未选择原因标签时，部分退款审核通过提交失败，并出现明确提示。
3. 原因标签选项完整且文案为：商品缺货、商品破损、运费差额、客服补偿、其他。
4. 审核通过后，后台详情页展示原因标签和标签操作人。
5. 后台退款列表可按任一原因标签筛选，筛选结果只包含对应标签记录。
6. 后台退款列表默认全部查询时，历史无标签退款单仍正常出现。
7. 退款导出报表包含“部分退款原因标签”字段。
8. 按原因标签筛选后导出，导出数据范围与列表筛选一致。
9. 历史退款单不补录，详情展示为空态，导出字段为空。
10. 无售后审核权限的用户不能通过本需求获得审核或标签写入能力。
11. C 端页面、C 端接口、C 端通知均不展示原因标签和标签操作人。
12. 已审核通过标签修改能力不存在于本期交付范围。

## 11. 测试关注点

- 部分退款审核通过：选择每个标签均可成功。
- 部分退款审核通过：不选标签失败。
- 非部分退款流程：不因该字段产生阻塞。
- 列表筛选：五个标签分别筛选准确。
- 列表默认查询：历史无标签单据仍存在。
- 详情展示：新单展示标签和操作人，历史单显示 `--`。
- 导出：新增字段列位置稳定，筛选和空值表现正确。
- 权限：仅现有售后审核权限用户可写入。
- C 端：相关接口响应和页面无新增字段泄露。

## 12. 发布与回滚

发布建议：

- 先发布兼容空值的数据结构变更。
- 再发布后端接口校验、查询、导出能力。
- 最后发布后台前端审核控件和筛选展示。

回滚建议：

- 若前端回滚，后端保留空字段不影响历史功能。
- 若后端回滚，前端需同步隐藏原因标签提交和筛选入口。
- 已写入的标签数据可保留，不影响既有退款单状态。

## 13. 复核结论

前端复核结论：认可。PRD 已明确后台审核控件、筛选、详情空态、C 端不可见边界。

后端复核结论：认可。PRD 已明确可空字段、操作人记录、权限复用、导出字段、历史数据兼容和未定项边界。
