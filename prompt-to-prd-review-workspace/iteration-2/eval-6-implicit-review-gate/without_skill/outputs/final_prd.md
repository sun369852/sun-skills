# 后台公告发布功能 PRD

版本：v1.0  
日期：2026-04-24  
状态：草案  

## 1. 背景

运营需要在后台发布站内公告，用于向指定用户群体传达产品通知、活动提醒、服务变更等信息。首版公告仅在 Web 端站内信中心展示，不扩展到短信、邮件或 App Push。

## 2. 目标

1. 支持运营在后台创建、发布、查看和下线站内公告。
2. 支持按展示时间范围控制公告可见性，结束后自动下线。
3. 支持按目标用户类型投放公告：全部用户、新用户、付费用户。
4. 首版仅支持纯文本正文和链接，避免引入富文本图片能力带来的审核、存储和安全复杂度。

## 3. 非目标

1. 不支持短信、邮件、App Push 等站外或移动端触达。
2. 不支持富文本图片、视频、附件、复杂排版。
3. 不支持用户分群自定义规则或精细化人群圈选。
4. 不支持公告评论、点赞、转发等互动能力。
5. 审批流是否纳入首版待确认，当前不作为已确定范围。

## 4. 用户与角色

| 角色 | 说明 | 核心诉求 |
| --- | --- | --- |
| 运营人员 | 后台公告的主要使用者 | 快速发布公告、控制投放人群和展示时间、必要时手动下线 |
| Web 用户 | 在站内信中心接收公告 | 能看到与自己匹配且仍在展示期内的公告 |
| 管理员 | 具备后台配置或管理权限的人员 | 管理公告权限、追踪公告状态 |

## 5. 业务范围

### 5.1 发布端

后台公告管理模块。

### 5.2 展示端

Web 端站内信中心。

### 5.3 目标用户类型

| 类型 | 说明 |
| --- | --- |
| 全部用户 | 所有 Web 登录用户可见 |
| 新用户 | 符合平台“新用户”定义的用户可见 |
| 付费用户 | 当前具备付费身份或有效付费权益的用户可见 |

“新用户”和“付费用户”的具体判定口径应复用现有用户标签或用户身份体系；如现有系统无统一口径，需要在开发前确认。

## 6. 功能需求

### 6.1 公告列表

后台应提供公告列表，支持运营查看已创建公告。

列表字段：

| 字段 | 说明 |
| --- | --- |
| 公告标题 | 展示公告标题 |
| 目标用户类型 | 全部用户、新用户、付费用户 |
| 展示开始时间 | 公告开始展示的时间 |
| 展示结束时间 | 公告结束展示的时间 |
| 状态 | 草稿、待展示、展示中、已下线、已结束 |
| 创建人 | 创建公告的后台账号 |
| 创建时间 | 公告创建时间 |
| 更新时间 | 最近一次编辑或状态变更时间 |

基础操作：

1. 新建公告。
2. 查看公告详情。
3. 编辑未开始展示的公告。
4. 手动下线未结束且未下线的公告。

### 6.2 新建公告

运营可以在后台新建公告。

表单字段：

| 字段 | 类型 | 必填 | 规则 |
| --- | --- | --- | --- |
| 标题 | 单行文本 | 是 | 建议 1-60 字；前后空格自动去除 |
| 正文 | 多行文本 | 是 | 支持纯文本和链接；不支持图片、附件、复杂富文本 |
| 展示开始时间 | 日期时间 | 是 | 不得晚于展示结束时间 |
| 展示结束时间 | 日期时间 | 是 | 必须晚于展示开始时间 |
| 目标用户类型 | 枚举 | 是 | 全部用户、新用户、付费用户三选一 |

链接要求：

1. 正文中允许出现 URL。
2. URL 应在 Web 端站内信中心可点击。
3. 链接打开方式按 Web 端现有链接规范执行；如无规范，默认新窗口打开。
4. 需要对链接协议进行安全限制，至少禁止 `javascript:` 等高风险协议。

### 6.3 发布公告

公告保存后进入可展示状态：

1. 如果当前时间早于展示开始时间，状态为“待展示”。
2. 如果当前时间处于展示时间范围内，状态为“展示中”。
3. 如果当前时间已晚于展示结束时间，公告不得发布，需提示运营调整时间。

如后续确认需要审批流，则发布动作可能变更为“提交审批”，详见“待确认事项”。

### 6.4 自动下线

公告到达展示结束时间后应自动下线，不再在 Web 端站内信中心展示。

要求：

1. 自动下线应基于服务端时间判断。
2. 结束时间到达后，公告状态应从“展示中”或“待展示”流转为“已结束”。
3. 已结束公告不可重新进入展示中；如需再次展示，应创建新公告或复制后重新发布。

### 6.5 手动下线

运营可以手动下线未结束公告。

规则：

1. 状态为“待展示”或“展示中”的公告可以手动下线。
2. 手动下线后状态变为“已下线”。
3. 已下线公告不再在 Web 端站内信中心展示。
4. 手动下线操作需要二次确认，避免误操作。
5. 手动下线后是否允许重新发布，首版建议不允许；如需再次发布，应创建新公告或复制后重新发布。

### 6.6 Web 端展示

Web 端站内信中心展示公告。

展示条件：

1. 用户已登录 Web 端。
2. 公告状态为“展示中”。
3. 当前服务端时间处于公告展示开始时间和展示结束时间之间。
4. 用户类型匹配公告目标用户类型。

展示内容：

| 字段 | 说明 |
| --- | --- |
| 标题 | 公告标题 |
| 正文 | 公告纯文本正文，正文中的安全链接可点击 |
| 发布时间或展示开始时间 | 用于用户识别公告时效 |

展示排序：

1. 默认按展示开始时间倒序。
2. 如同一时间有多条公告，按创建时间倒序。

### 6.7 状态流转

| 当前状态 | 触发动作 | 目标状态 | 说明 |
| --- | --- | --- | --- |
| 草稿 | 发布且开始时间在未来 | 待展示 | 等待开始展示 |
| 草稿 | 发布且当前处于展示范围内 | 展示中 | 立即展示 |
| 待展示 | 到达展示开始时间 | 展示中 | 自动展示 |
| 待展示 | 手动下线 | 已下线 | 不再展示 |
| 展示中 | 到达展示结束时间 | 已结束 | 自动下线 |
| 展示中 | 手动下线 | 已下线 | 运营主动下线 |

如果首版不实现草稿能力，则创建并发布可合并为一个动作，但仍需要保留“待展示、展示中、已下线、已结束”的状态语义。

## 7. 权限与审计

1. 仅具备公告管理权限的后台账号可创建、编辑、发布和下线公告。
2. 公告创建、编辑、发布、手动下线应记录操作人、操作时间和关键变更内容。
3. 手动下线应记录下线原因或至少记录操作备注；是否必填可按后台现有规范确认。

## 8. 数据需求

公告核心字段建议如下：

| 字段 | 说明 |
| --- | --- |
| id | 公告唯一标识 |
| title | 标题 |
| content | 正文 |
| target_user_type | 目标用户类型 |
| display_start_at | 展示开始时间 |
| display_end_at | 展示结束时间 |
| status | 公告状态 |
| created_by | 创建人 |
| created_at | 创建时间 |
| updated_by | 最近更新人 |
| updated_at | 最近更新时间 |
| offline_by | 手动下线人 |
| offline_at | 手动下线时间 |
| offline_reason | 手动下线原因，可选 |

## 9. 边界规则

1. 展示时间范围以服务端时间为准，避免客户端时间不一致导致展示异常。
2. 展示结束时间到达后，Web 端不应继续返回该公告。
3. 正文为空、标题为空、结束时间不晚于开始时间时，不允许保存或发布。
4. 公告正文中的链接需要做安全过滤和转义，避免 XSS。
5. 同一用户可看到多条匹配公告，首版不做频控或已读强制过滤；是否支持已读状态可按站内信中心现有能力处理。

## 10. 验收标准

1. 运营可以在后台创建包含标题、正文、展示时间范围、目标用户类型的公告。
2. 公告在展示开始时间前不出现在 Web 端站内信中心。
3. 公告在展示时间范围内，仅对匹配目标用户类型的用户展示。
4. 公告到达展示结束时间后自动下线，Web 端不再展示。
5. 运营可以手动下线未结束公告，下线后 Web 端不再展示。
6. 正文支持纯文本和安全链接展示，不支持上传或展示富文本图片。
7. 短信、邮件、App Push 不会因公告发布被触发。
8. 审批流作为待确认事项记录，不阻塞首版 PRD 内容。

## 11. 埋点与指标

首版建议至少保留以下运营观察指标：

| 指标 | 说明 |
| --- | --- |
| 公告创建数 | 指定周期内创建的公告数量 |
| 公告展示中数量 | 当前展示中的公告数量 |
| 公告手动下线数 | 被运营主动下线的公告数量 |
| 公告曝光量 | Web 站内信中心中公告被用户看到的次数，如现有埋点支持 |
| 公告链接点击量 | 正文链接被点击的次数，如现有埋点支持 |

曝光和点击埋点如果站内信中心已有统一机制，应复用现有能力；首版不强制新增复杂分析看板。

## 12. 待确认事项

1. 公告是否需要审批流：首版先作为待确认事项记录。若需要审批，需要补充审批角色、审批状态、驳回流程、发布权限和审计要求。
2. “新用户”的判定口径：按注册时间、首次登录时间、是否完成某关键行为，或复用现有用户标签。
3. “付费用户”的判定口径：按当前订阅有效、历史付费、会员身份或订单状态。
4. 手动下线是否必须填写下线原因。
5. 已下线公告是否允许重新发布；首版建议不允许。
6. 是否需要公告已读/未读状态；如站内信中心已有能力，建议复用。

## 13. 依赖

1. 后台权限体系。
2. 用户类型或用户标签识别能力。
3. Web 端站内信中心展示能力。
4. 服务端定时任务或查询时状态判断能力，用于自动下线。
5. 文本链接安全处理与 XSS 防护能力。
