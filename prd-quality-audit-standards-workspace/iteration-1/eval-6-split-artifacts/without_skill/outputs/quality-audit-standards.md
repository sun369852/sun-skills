# PRD 质量审核标准

## 目的

为大型 PRD 提供可复用的质量审核标准，覆盖 Web、后台任务、外部支付回调、数据迁移和审计报表。主文档只保留审核原则、范围和通过门槛；细化检查项、上下文记录和报告格式拆分到独立文件。

## 拆分文件

- [audit-checks.json](./audit-checks.json)：结构化检查项，适合导入审核工具或逐项勾选。
- [clean-context-review-record.md](./clean-context-review-record.md)：审核前的上下文清洁记录，避免遗漏依赖、口径和历史约束。
- [audit-report-template.md](./audit-report-template.md)：审核报告模板，用于输出结论、风险、阻塞项和整改建议。

## 审核范围

1. Web 体验：页面流程、状态、权限、错误提示、空态、加载态、埋点和可访问性。
2. 后台任务：触发条件、调度频率、幂等、重试、失败补偿、监控告警和人工介入路径。
3. 外部支付回调：验签、幂等、订单状态机、异常回调、重复通知、金额币种校验和对账。
4. 数据迁移：迁移范围、兼容策略、回滚方案、数据校验、灰度批次和业务停机窗口。
5. 审计报表：指标口径、查询权限、导出限制、数据时效、留痕要求和合规约束。

## 质量分级

- P0：存在上线阻断风险，必须修复后才能进入开发或发布。
- P1：存在明确业务、资金、数据或合规风险，必须在当前版本闭环。
- P2：影响体验、可维护性或运营效率，需排期处理并标注责任人。
- P3：优化建议，不阻塞当前版本。

## 通过门槛

PRD 通过审核需同时满足：

1. `audit-checks.json` 中所有 P0 检查项均为通过或不适用，且不适用原因明确。
2. Web、后台任务、支付回调、数据迁移、审计报表五类范围均有明确需求边界。
3. 涉及资金、权限、数据迁移和审计留痕的 P1 问题已有责任人、修复方案和验收方式。
4. `clean-context-review-record.md` 已记录审核依据、缺失资料和待确认问题。
5. `audit-report-template.md` 已输出最终结论，且阻塞项与检查项编号可追溯。

## 审核流程

1. 清洁上下文：使用 [clean-context-review-record.md](./clean-context-review-record.md) 记录 PRD 版本、关联材料和待确认范围。
2. 逐项检查：使用 [audit-checks.json](./audit-checks.json) 对五类功能范围执行检查。
3. 汇总风险：按 P0-P3 分级整理问题，标注影响面、证据、责任人和截止时间。
4. 输出报告：使用 [audit-report-template.md](./audit-report-template.md) 形成最终审核结论。

## 审核原则

- 先判断需求是否可验收，再判断方案是否完整。
- 资金、权限、数据、合规相关内容优先级高于普通体验问题。
- 每个风险必须能追溯到 PRD 原文、缺失信息或明确假设。
- 不把实现方案写成审核结论；结论应说明风险和验收标准。
- 对“不适用”项必须记录理由，不能用空白代替判断。
