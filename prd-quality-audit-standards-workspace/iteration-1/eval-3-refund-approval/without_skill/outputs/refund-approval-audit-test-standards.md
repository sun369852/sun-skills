# 支付退款审批 PRD 审核测试标准

## 输入与假设

- 输入来源：用户描述的支付退款审批 PRD，高风险场景。
- 未读取禁用目录：`D:\sun-skills\prd-quality-audit-standards`。
- 假设系统包含退款申请、审批、执行第三方退款、结果回写、通知、审计日志等核心流程。
- 本文目标：给开发完成后的测试审核 reviewer 使用，判断功能是否达到可验收的测试覆盖标准。

## 三轮干净上下文复核

### 第一轮：权限与审批状态流转

关注点：
- 只有具备退款申请权限的角色可以发起退款。
- 只有具备审批权限的角色可以审批，且审批人不能绕过业务约束。
- 退款审批状态必须单向、可解释、可追踪。
- 非法状态跳转必须被拒绝并记录。

审核标准：
- 覆盖不同角色的访问控制测试：申请人、审批人、财务、客服、管理员、无权限用户。
- 覆盖越权访问测试：直接调用接口、篡改订单号、篡改退款单号、跨商户/跨组织访问。
- 覆盖审批人与申请人是否允许为同一人的规则；若 PRD 未明确，应标记为阻塞问题。
- 覆盖状态流转矩阵测试，至少包括：
  - `待提交 -> 待审批`
  - `待审批 -> 审批通过`
  - `待审批 -> 审批拒绝`
  - `审批通过 -> 退款处理中`
  - `退款处理中 -> 退款成功`
  - `退款处理中 -> 退款失败`
  - `退款失败 -> 重试中/人工处理/关闭`，以 PRD 定义为准
- 覆盖非法流转测试，至少包括：
  - 已拒绝退款再次审批通过
  - 已成功退款再次发起第三方退款
  - 退款处理中重复审批
  - 已关闭单据再次进入处理中
- 覆盖并发审批测试：两个审批人同时操作同一退款单时，只能有一个最终有效结果。
- 覆盖审批意见、审批时间、审批人、审批结果的完整记录。

第一轮结论：
- 权限与状态流转是高风险主线，必须形成明确状态机和权限矩阵。
- 若 PRD 缺少角色权限表、状态机、非法跳转规则，则不能直接进入开发后测试审核。

### 第二轮：退款幂等、第三方失败与数据一致性

关注点：
- 同一退款请求不会因为重复提交、网络重试、回调重复而重复退款。
- 第三方退款失败、超时、处理中、回调乱序时，系统状态保持一致。
- 本地退款单、订单、支付流水、第三方退款流水之间关系清晰。

审核标准：
- 覆盖退款申请幂等测试：
  - 同一订单、同一支付流水、同一退款金额重复申请时的处理规则。
  - 用户重复点击提交按钮不会生成多笔有效退款。
  - 接口重复请求使用幂等键或业务唯一约束防重。
- 覆盖退款执行幂等测试：
  - 审批通过后重复触发退款执行，不会向第三方发起多笔有效退款。
  - 服务重启、任务重试、消息重复消费后仍只产生一笔退款。
- 覆盖第三方回调幂等测试：
  - 重复成功回调只更新一次成功状态。
  - 成功回调后又收到失败/处理中回调，不得覆盖最终成功结果。
  - 失败回调后收到成功回调时，按第三方权威状态和 PRD 规则处理。
- 覆盖第三方失败场景：
  - 第三方同步返回失败。
  - 第三方同步返回处理中，后续异步失败。
  - 第三方接口超时，本地状态不得直接标记成功。
  - 第三方接口不可用，系统进入可重试或人工处理状态。
  - 第三方返回错误码不可识别时，系统不得丢失退款单。
- 覆盖金额一致性测试：
  - 部分退款、全额退款、重复部分退款后的累计金额不得超过可退金额。
  - 手续费、优惠券、积分、组合支付等退款金额规则，如 PRD 涉及必须覆盖。
  - 退款失败后不应扣减可退余额；退款成功后必须准确扣减。
- 覆盖事务一致性测试：
  - 本地退款单创建成功但审批记录写入失败。
  - 第三方退款成功但本地更新失败后的补偿。
  - 本地标记成功但通知失败时，不影响退款最终状态。

第二轮结论：
- 幂等与第三方失败处理必须作为测试审核的硬性门槛。
- 若 PRD 未定义幂等键、唯一约束、第三方回调优先级、失败重试策略，应要求补充。

### 第三轮：审计日志、通知与可运营性

关注点：
- 高风险资金操作必须具备完整审计链路。
- 用户、审批人、财务、运营等相关方需要收到准确通知。
- 测试审核不仅验证功能可用，还要验证问题可追踪、可恢复、可解释。

审核标准：
- 覆盖审计日志完整性测试：
  - 退款申请创建日志。
  - 审批通过/拒绝日志。
  - 退款执行发起日志。
  - 第三方请求与响应摘要日志。
  - 第三方回调日志。
  - 人工重试、人工关闭、人工改判日志。
- 覆盖审计日志字段测试，至少包括：
  - 操作人、角色、操作时间、操作对象、前置状态、后置状态、操作结果、失败原因、请求来源。
  - 金额、币种、订单号、支付流水号、退款单号、第三方退款流水号。
- 覆盖日志不可抵赖测试：
  - 审计日志不能被普通业务角色修改或删除。
  - 敏感信息脱敏，避免记录银行卡、完整证件号、完整支付凭证等不必要明文。
- 覆盖通知场景：
  - 退款申请提交后通知审批人。
  - 审批通过/拒绝后通知申请人或相关业务方。
  - 退款成功/失败后通知用户或业务系统。
  - 第三方退款异常进入人工处理时通知运营/财务。
- 覆盖通知准确性：
  - 通知内容中的金额、订单号、退款状态、失败原因与系统记录一致。
  - 通知失败不应影响退款主状态。
  - 重复回调或重复任务不会造成重复通知，或重复通知符合 PRD 明确规则。
- 覆盖运营可见性：
  - 能按退款单号、订单号、支付流水号、第三方流水号查询。
  - 能区分审批拒绝、退款失败、第三方处理中、人工处理等状态。
  - 能看到失败原因和下一步处理建议。

第三轮结论：
- 审计日志与通知是资金系统测试审核必选项。
- 若 PRD 只描述退款成功路径，未覆盖异常可追踪性，则审核标准不通过。

## 合并后的审核测试标准

### 1. PRD 完整性门槛

PRD 至少应明确：
- 角色权限矩阵。
- 退款审批状态机。
- 每个状态的进入条件、退出条件、可操作角色。
- 幂等规则和唯一约束。
- 第三方退款接口成功、失败、处理中、超时、回调乱序的处理规则。
- 审计日志字段。
- 通知对象、通知时机、通知内容和通知失败处理。
- 人工处理、重试、关闭、补偿机制。

缺少以上任一高风险主项时，应退回 PRD 补充，不建议进入开发后测试审核。

### 2. 必测测试域

| 测试域 | 审核要求 | 通过标准 |
| --- | --- | --- |
| 权限控制 | 覆盖角色、组织、商户、接口越权 | 无权限用户无法查看、创建、审批、执行或修改退款 |
| 状态流转 | 覆盖合法与非法状态跳转 | 状态只能按 PRD 状态机变化，非法跳转被拒绝 |
| 审批并发 | 覆盖多人同时审批 | 只产生一个最终审批结果，其他请求明确失败 |
| 退款幂等 | 覆盖重复提交、重复执行、重复回调 | 不产生重复退款，不重复扣减可退金额 |
| 金额一致性 | 覆盖全额、部分、累计退款 | 累计退款不超过可退金额，失败不扣减成功额度 |
| 第三方失败 | 覆盖失败、超时、处理中、乱序回调 | 状态准确，可重试或可人工处理，不丢单 |
| 审计日志 | 覆盖全链路资金操作 | 每个关键动作可追踪，字段完整且敏感信息脱敏 |
| 通知 | 覆盖申请、审批、成功、失败、异常 | 通知对象和内容正确，通知失败不污染主流程 |
| 数据一致性 | 覆盖本地单据、订单、支付流水、第三方流水 | 关键记录可关联，异常后可恢复 |
| 可运营性 | 覆盖查询、筛选、失败原因、人工处理 | 运营/财务能定位并处理异常退款 |

### 3. 阻塞级审核问题

出现以下任一问题，应判定不能用于开发完成后的测试审核：
- 未定义审批状态机。
- 未定义退款幂等策略。
- 未定义第三方退款失败、超时、回调重复或乱序处理。
- 未定义权限边界，尤其是跨商户/跨组织访问。
- 未定义审计日志字段或关键操作不记日志。
- 未定义退款成功后的金额一致性规则。
- 未定义异常退款的人工处理或补偿路径。

### 4. 建议测试证据

开发完成后测试审核时，应要求提供：
- 权限矩阵测试结果。
- 状态流转测试用例与执行记录。
- 幂等测试证据，包括重复请求、重复任务、重复回调。
- 第三方失败模拟测试结果。
- 并发审批测试结果。
- 审计日志样例。
- 通知发送记录与失败重试/降级记录。
- 数据库关键表记录截图或导出样例。
- 异常退款人工处理演示记录。

## 最终判断

结论：当前审核测试标准可以作为开发完成后的测试审核基线，但前提是 PRD 已补齐权限矩阵、状态机、幂等策略、第三方异常处理、审计日志和通知规则。

若现有 PRD 仅有高层需求描述，缺少上述规则，则不能直接用于测试审核，应先退回补充 PRD，再基于补充后的 PRD 细化测试用例。
