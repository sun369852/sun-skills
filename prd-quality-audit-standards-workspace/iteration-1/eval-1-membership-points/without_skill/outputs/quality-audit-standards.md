# 会员积分 PRD 代码开发完成后测试审核标准

## 1. 审核范围

本标准用于代码开发完成后的测试审核，覆盖会员积分 PRD 中已明确的功能：

- 消费发放积分
- 退款冲正积分
- 积分过期
- 管理员调整积分
- 会员积分流水

不审核 PRD 未提出的新能力，例如积分兑换、会员等级、营销活动倍率、跨账户转赠等。

## 2. 总体验收标准

开发完成后，测试审核应确认以下结论全部成立：

- 积分余额、可用积分、冻结或待生效积分的计算口径与 PRD 一致。
- 每一次积分变动都有对应流水，流水金额、方向、业务类型、关联单据、发生时间可追溯。
- 消费、退款、过期、管理员调整之间不会造成重复发放、重复扣减、负数异常或流水缺失。
- 同一业务单据重复请求时结果幂等，不产生额外积分变动。
- 并发场景下积分余额与流水汇总结果一致。
- 用户端与管理员端展示的余额和流水记录一致，差异仅来自权限或字段脱敏要求。

## 3. 功能审核标准

### 3.1 消费发放积分

测试审核点：

- 订单满足 PRD 规定的发放条件时，系统按正确规则发放积分。
- 未支付、已取消、无效订单、退款完成订单不应发放积分。
- 积分计算基数正确，排除 PRD 规定不参与积分计算的金额或项目。
- 小数、舍入、最低发放门槛按 PRD 规则处理。
- 同一订单重复触发发放逻辑时，只发放一次积分。
- 发放成功后，会员积分余额增加，积分流水新增一条发放记录。
- 流水中的业务类型、积分数量、订单号、会员标识、发生时间、操作来源正确。

通过标准：

- 正常订单发放结果与预期一致。
- 不符合条件的订单不产生积分和流水。
- 重复请求、消息重投、接口重试不导致重复发放。

### 3.2 退款冲正积分

测试审核点：

- 全额退款时，系统按 PRD 冲正已发放积分。
- 部分退款时，系统按 PRD 比例或金额规则冲正积分。
- 多次部分退款时，累计冲正积分不超过原发放积分。
- 未发放积分的订单退款时，不应产生冲正积分。
- 退款冲正后，会员积分余额正确减少。
- 用户积分不足以冲正时，处理方式符合 PRD，例如扣至负数、限制退款、记录欠扣或其他明确规则。
- 退款冲正流水与原订单、退款单、原积分发放流水存在可追溯关联。
- 同一退款单重复处理时，不重复冲正。

通过标准：

- 全额、部分、多次退款的余额和流水均正确。
- 冲正不会超过原发放积分。
- 幂等和异常余额场景处理符合 PRD。

### 3.3 积分过期

测试审核点：

- 积分有效期计算符合 PRD，包括起算时间、到期时间、自然日或固定周期规则。
- 到期积分在正确时间被识别并过期。
- 未到期积分不会被提前过期。
- 已使用、已冲正、已过期的积分不会再次过期。
- 多批次积分存在不同有效期时，系统按批次正确过期。
- 过期任务重复执行时，不重复扣减。
- 过期后会员余额减少，并生成过期流水。
- 过期流水应包含会员标识、过期积分数量、积分批次或来源、过期时间、业务类型。

通过标准：

- 到期边界前后结果准确。
- 批次维度余额、总余额、流水汇总一致。
- 定时任务或补偿任务重复执行保持幂等。

### 3.4 管理员调整积分

测试审核点：

- 管理员可按 PRD 进行增加或扣减积分。
- 调整原因、操作人、操作时间、会员标识、调整数量必填并落库。
- 扣减积分时，余额不足处理符合 PRD。
- 超出 PRD 限制的调整数量、非法会员、无权限管理员操作应被拒绝。
- 调整成功后，会员余额立即更新。
- 调整成功后，生成管理员调整流水。
- 审计信息可追溯到具体管理员和调整原因。

通过标准：

- 合法调整成功且余额、流水、审计信息一致。
- 非法调整不改变余额，不生成成功流水。
- 权限控制符合 PRD 或系统既有权限模型。

### 3.5 会员积分流水

测试审核点：

- 所有积分变动均有流水，包括消费发放、退款冲正、积分过期、管理员调整。
- 流水方向正确，增加为正向，扣减、冲正、过期为负向或 PRD 指定方向。
- 流水积分数量与实际余额变动一致。
- 流水可按会员、时间、业务类型、关联单据查询。
- 流水排序稳定，默认排序符合 PRD 或产品约定。
- 流水详情字段完整，能说明积分从何而来、因何变化。
- 失败、取消、重复请求不生成成功流水。
- 后台流水与用户端流水在共享字段上保持一致。

通过标准：

- 流水总和与会员当前余额在同一统计口径下可对账。
- 任一余额变化均能追溯到至少一条有效流水。
- 查询、筛选、分页不会遗漏或重复展示记录。

## 4. 跨功能一致性审核标准

测试审核点：

- 消费发放后发生退款，积分最终余额等于发放积分减去冲正积分。
- 消费发放后积分到期，再发生退款时，不出现重复扣减或超过原发放积分的冲正。
- 管理员调整不破坏消费、退款、过期流水的业务关联。
- 积分过期任务与退款冲正、管理员调整并发时，余额不出现计算丢失。
- 会员积分余额等于所有有效流水按方向汇总后的结果，除非 PRD 明确存在冻结、待生效、欠扣等独立口径。

通过标准：

- 任意组合场景下，余额、批次、流水三者一致。
- 并发和重试后不会出现重复流水、漏流水、余额错算。

## 5. 数据与接口审核标准

测试审核点：

- 积分变动接口具备幂等标识或等价防重机制。
- 核心积分变动操作在事务内完成，余额和流水不出现一方成功一方失败。
- 关联订单号、退款单号、管理员操作记录等外部标识保存准确。
- 关键字段类型和精度满足积分计算要求。
- 时间字段使用统一时区和统一存储口径。
- 异常失败后可重试，重试不造成重复积分变动。

通过标准：

- 数据库记录与接口返回一致。
- 失败恢复后数据仍可对账。
- 核心表不存在孤立流水或无流水余额变更。

## 7. 缺陷判定标准

以下问题应判定为阻断级或高优先级缺陷：

- 积分余额计算错误。
- 积分发放、冲正、过期、调整出现重复处理。
- 积分流水缺失或与余额变动不一致。
- 用户积分被错误扣减或错误增加。
- 并发或重试导致数据不一致。
- 管理员无权限调整成功。
- 退款冲正超过原发放积分。

以下问题可根据影响范围判定为中低优先级缺陷：

- 流水展示字段缺失但底层数据完整。
- 查询筛选条件不完整但不影响积分正确性。
- 文案、排序、时间展示格式与 PRD 轻微不一致。

## 8. 三轮干净上下文复核记录

### 第一轮：需求覆盖复核

复核目标：确认测试审核标准覆盖 PRD 已列功能，不引入无关产品范围。

复核结论：

- 已覆盖消费发放积分、退款冲正、积分过期、管理员调整、会员积分流水。
- 未将积分兑换、会员等级、营销倍率等未给定功能纳入必测范围。
- 每个功能均包含测试审核点和通过标准。

### 第二轮：数据一致性复核

复核目标：确认标准能发现余额、流水、批次、关联单据之间的不一致。

复核结论：

- 已要求所有积分变动必须生成流水。
- 已要求余额与流水汇总可对账。
- 已覆盖事务一致性、幂等、并发和重试。
- 已覆盖退款、过期、调整之间的组合场景。

### 第三轮：可执行性复核

复核目标：确认标准可被测试人员直接转化为测试用例和缺陷判定。

复核结论：

- 已提供必测场景编号、场景描述和预期审核结论。
- 已明确阻断级或高优先级缺陷类型。
- 审核点均可通过接口、数据库、后台页面或用户端页面验证。
- 对 PRD 未明确的处理规则，均要求按 PRD 口径验证，避免自行新增规则。

## 9. 最终审核通过条件

代码开发完成后的测试审核通过，需要同时满足：

- 第 3 节全部功能审核标准通过。
- 第 4 节跨功能一致性审核标准通过。
- 第 5 节数据与接口审核标准通过。
- 第 6 节必测场景全部执行完成，未遗留阻断级或高优先级缺陷。
- 所有积分余额变更均可被流水解释，所有流水均可追溯到具体业务来源或管理员操作。
